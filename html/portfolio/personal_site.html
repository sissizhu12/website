              portfolio website

              portfolio website

              introduction


              When creating this portfolio website from scratch, my initial goal was to leverage the full creative freedom to build a unique portfolio. I also wanted to engage with and explore website navigation, conception, and accessibility more deeply.


              my role


              I conducted all research, designing, coding, and gathering feedback, which ultimately allowed me to see the connections between each role and brought me new insights as a website designer. Most importantly, I learned to understand the worlds of design and code as collaborative rather than independent.


              the challenge


              My main goal was to avoid using external plugins, website templates, or frameworks. The intention was to deepen my knowledge of the building blocks of web applications: HTML and CSS. I also set internal deadlines to stick to, which was helpful for the longevity of the project!

              how did I get started?
              (research to build a persona)


              I knew that generally speaking, viewers of my portfolio would be job recruiters, HR, or even potential clients looking to gauge my skillset.

              To better understand their needs and pain points when navigating portfolio websites, I did secondary research through articles, videos, and interviews. I then assembled a user persona for the most frequent kind of visitor: a recruiter or HR viewing my portfolio website.

              exploring webflow with card sorting


              I then asked myself how and what content should be organized on the website for easy content navigation. To explore options and see meaningful connections, I started by sorting website content into groups.

              This exercise also made me consider what content was most and least important to showcase.

              When organizing the content into a more detailed user flow chart, I realized that I would need to make sure that users could quickly navigate and differentiate the pages to find what they're looking for.

              Because the most popular website entry points would be the home page or the portfolio page, I needed to make sure those pages especially had a visually salient and consistent navigation bar.

              This insight ultimately helped guide my initial designs.

              when in doubt, sketch it out


              To optimise the website layout responsiveness, I realized two key guidelines:


              1. The layout of photos, videos, and embeds must flexibly scale with similar screen ratios

              2. Webpage navigation can’t overly rely on vertical scrolling

              Through explorative sketches, I found a solution of using different navigation bars and content layouts depending on screen size.

              'test driving' the visual inspiration


              For visual inspiration, I drew on images from my short film “Dream” due to its vibrant and dreamy colours.

              I wanted to use the images as a theme to show my introspective personality, so I created initial wireframes in Figma based on the colour palette and tone.

              refining my coding skills


              After getting a clear visual plan, I built out the designs in code. Throughout the process, I practiced design skills and key coding skills:


            • Using Git for version control and regular commits (gh/sissizhu12/website)

            • Organizing files and structuring code with comments

            • Setup of automatic website deployment on GitHub workflows

            • HTML and CSS strategies for responsive webpages

              • reflection


              Working both roles of coding and design taught me how design and code depend on each other throughout the process of building digital tools.


              This project made me appreciate both sides more, as I often moved back and forth between designing and coding, giving me new insights from one role to support the other. For instance, I often returned to sketching when coding to help visualize, debug, or adjust design plans if they didn't practically pan out the way I anticipated. Similarly, I learned what design elements are crucial to have planned out before starting to code versus what kinds of details are more flexible and can be adjusted later on.
